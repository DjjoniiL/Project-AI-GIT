# Точка продолжения работы — читать в начале новой сессии

Этот файл — сводка для быстрого восстановления контекста в новой сессии Claude Code. Источник истины по архитектуре/требованиям — `specification.md`, второй обязательный документ — `README.txt`. Этот файл даёт быстрый снимок "что сделано / что дальше" без необходимости перечитывать историю чата.

## Что за проект

MVP веб-конструктора заказов на печать одежды (футболки/худи/свитшоты/лонгсливы), встраиваемого вкладкой в карточку сделки Битрикс24 через платформу **VibeCode** (не классический REST+OAuth2, см. `specification.md` раздел 2). Фронтенд собирает параметры заказа + макет принта, бэкенд создаёт/обновляет сделку и грузит файл на Диск Битрикс24.

## Живой тестовый сервер (galaxyApp)

- id `801496ff-f700-4dd0-9bec-571adbc7efcb`, appUrl `https://app-2a2f90ace69d.vibecode.bitrix24.tech`, accessPolicy `OWNER_ONLY`.
- Для проверки живьём без браузера — минтить временный токен: `POST /v1/infra/servers/801496ff-f700-4dd0-9bec-571adbc7efcb/access-tokens {"mode":"api-bearer"}` (личным `vibe_api_` ключом), потом `Authorization: Bearer <token>` на запросах к appUrl. Токен живёт 10 минут (`jwtExpiresAt`), можно перевыпустить через `POST .../access-tokens/:tokenId/refresh` не тратя лимит минта.
- Логи: `GET /v1/infra/servers/801496ff-f700-4dd0-9bec-571adbc7efcb/logs?lines=100`.
- ✅ **С этой сессии фронтенд тоже деплоится вместе с бэкендом** — деплой-архив теперь включает **предсобранный локально** `apps/frontend/dist` (не исходники фронтенда, сервер его не собирает — экономит время/память на контейнере с лимитом 512 МБ). Полный актуальный контракт `POST /deploy` (см. `apps/backend/src/app.ts` для serving-логики, `README.txt`/`specification.md` раздел 8 для статуса):
  ```json
  {
    "source": { "content": "<base64 tar.gz архива>" },
    "runtime": "node20",
    "install": "npm install && npm run build --workspace packages/shared-types && npm run build --workspace apps/backend",
    "preStart": "apt-get update -qq && apt-get install -y -qq libglib2.0-0 libnss3 libatk1.0-0 libatk-bridge2.0-0 libcups2 libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 libxfixes3 libxrandr2 libgbm1 libasound2 libpango-1.0-0 libcairo2",
    "start": "node apps/backend/dist/index.js",
    "port": 3000,
    "healthPath": "/api/health",
    "env": { "PORT": "3000", "VIBE_APP_KEY": "<из .env>", "VIBE_LAYOUT_FOLDER_ID": "<из .env>" }
  }
  ```
  Содержимое архива (tar.gz): корневые `package.json`+`package-lock.json`+`tsconfig.base.json`, `packages/shared-types/{package.json,tsconfig.json,src/}`, `apps/backend/{package.json,tsconfig.json,src/}`, **`apps/frontend/dist/` целиком** (без `apps/frontend/package.json` и остального исходника фронтенда — директория без `package.json` внутри `apps/*` просто игнорируется npm-воркспейсами при `npm install`, проверено эмпирически, ошибок не даёт). `env` нужно передавать **при каждом деплое** — платформа перегенерирует `.env` на сервере с нуля, старые значения не сохраняются между деплоями, если их не передать явно.
  - **`preStart` для Puppeteer — теперь обязательная часть каждого деплоя** (раньше апт-пакеты ставились вручную через `/exec` на существующем сервере, теперь `puppeteer` — задекларированная зависимость `apps/backend`, и `npm install` в шаге `install` качает сам Chromium; без `preStart` с системными библиотеками реальный запуск `puppeteer.launch()` упадёт на «missing libglib-2.0.so.0» и т.п., как и раньше).
  - **Важное открытие про надёжность деплоя galaxy-приложений**: во время этой сессии несколько попыток `/deploy` подряд возвращали `502 GALAXY_HOST_UNREACHABLE` («Galaxy host ... is DISCONNECTED») или `409 GALAXY_APP_BUSY`/`EXEC_BUSY`, хотя `GET /v1/me` и `GET /v1/infra/servers/:id` в это же время отвечали нормально. Оказалось — деплой **реально шёл на сервере в фоне**, а мои HTTP-соединения обрывались раньше получения ответа (сеть пользователя нестабильна — см. заметку про Outline VPN выше, тот же класс проблемы). Ретраи `/deploy` в таком состоянии просто возвращали «занято» (это ожидаемо и безопасно — деплой блокирует сам себя локом на 15 минут, а не запускается параллельно ещё раз). Что сработало: подождать (внутри `until`-цикла в Bash-инструменте опрашивать `GET /v1/infra/servers/:id` на `blackholeStatus:"CONNECTED"`), затем **проверить реальным HTTP-запросом к `appUrl` через access-token**, а не доверять полю `status` в ответе `GET /v1/infra/servers/:id`.
  - **Поле `status` ненадёжно как индикатор готовности galaxy-приложения.** После полностью успешного деплоя (подтверждено живыми запросами: `/`, SPA-фолбэк, `/api/health`, `/api/order-pdf` — все отвечали правильно) `GET /v1/infra/servers/:id` продолжал показывать `"status":"sleeping"` (при этом `"blackholeStatus":"CONNECTED"`). Похоже, `status` для `GALAXY_APP` отражает что-то вроде биллингового/idle-состояния контейнера, а не «отвечает ли он на HTTP прямо сейчас» — **не полагаться на это поле для проверки успешности деплоя, всегда проверять реальным запросом к `appUrl`**.
  - Если снова увидите `GALAXY_HOST_UNREACHABLE`/`GALAXY_APP_BUSY` — не пересоздавать сервер и не паниковать: подождать 1–3 минуты, проверить статус, при необходимости повторить `/deploy` ровно один раз (лишние параллельные попытки просто получат `409` и ничего не сломают).
- Для передеплоя после новых правок в коде — сначала `npm run build --workspace apps/frontend` локально (обновит `apps/frontend/dist`), затем собрать архив как описано выше и `POST /v1/infra/servers/801496ff-f700-4dd0-9bec-571adbc7efcb/deploy` (не создавать новый сервер через `POST /v1/infra/servers` — это создаст второй, отдельный).
