## Что за проект

MVP веб-конструктора заказов на печать одежды (футболки/худи/свитшоты/лонгсливы), встраиваемого вкладкой в карточку сделки Битрикс24 через платформу **VibeCode** (не классический REST+OAuth2, см. `specification.md` раздел 2). Фронтенд собирает параметры заказа + макет принта, бэкенд создаёт/обновляет сделку и грузит файл на Диск Битрикс24.

## Что нужно сделать в первую очередь (Этап 1, по плану — specification.md раздел 8)

1. **Создать ключ `vibe_app_...`** через `POST /v1/apps {title, scopes:[...,"placement"], appUrl?}` личным ключом `vibe_api_...` (appUrl можно оставить пустым, добавить позже через `PATCH /v1/apps/:id`, когда появится реальный URL после деплоя). Сохранить `rawKey` — показывается один раз. Использовать только для живых запросов, не класть в файлы репозитория кроме `.env`.
2. **Создать реальные UF-поля** на портале через `POST /v1/userfields/deals` (11 полей из `specification.md` раздела 6) и подставить реальные коды в `packages/shared-types/src/index.ts` вместо плейсхолдеров `UF_CRM_PRODUCT_TYPE` и т.д. Стоит попробовать тип `file` для поля «Файл макета» напрямую на сделке (кандидат, не проверялся эмпирически) вместо текстового поля со ссылкой на Диск.
3. Написать бэкенд-логику (Express + TS, `apps/backend`), с тестами на каждую функцию (Vitest + supertest, `npm test` после каждой правки — обязательное требование проекта):
   - `POST /api/order` с валидацией входных данных (`multipart/form-data`, файл до 52 МБ).
   - identity-middleware: пробрасывает заголовок `X-Vibe-Authorization` из входящего запроса как `Authorization` в `GET /v1/me`, получает личность пользователя (см. `specification.md` раздел 2 — механизм Gateway-инжекции токена для встроенного в портал приложения).
   - `createOrUpdateDeal(dealData, dealId?)` — `POST /v1/deals` или `PATCH /v1/deals/{id}` в зависимости от наличия `dealId` (который приходит из `placement_options`, не из простого query-параметра — раздел 7).
   - `uploadFileToDisk(file, dealId)` — `POST /v1/files/upload`, JSON-тело с base64 `content`, обязательно проверять исходный размер файла (≤52 МБ) до кодирования в base64.
4. **Создать ключ `vibe_app_` и задеплоить бэкенд** на тестовый портал (`vibecode02.bitrix24.ru`), режим `galaxyApp` (см. `specification.md` раздел 2 — точный контракт `POST /v1/infra/servers`/`/deploy`). Заодно **эмпирически проверить поддержку Puppeteer/headless Chromium** (документация не подтверждает и не опровергает; ~512 МБ RAM в galaxy-контейнере — риск-сигнал; fallback `pdf-lib`/`@react-pdf/renderer` наготове).
5. Протестировать `POST /v1/placements/bind` живым запросом ключом `vibe_app_` и пройти реальный flow встраивания вкладки `CRM_DEAL_DETAIL_TAB` в карточку сделки на тестовом портале.

## Технический стек (зафиксирован, не пересматривать без явного запроса)

- Фронтенд: React 18 + TypeScript + Redux Toolkit + Ant Design + Vite.
- Бэкенд: Node.js + Express + TypeScript, деплой на BlackHole VibeCode — конкретно в режиме `galaxyApp` (общий хост, контейнер, ~512 МБ RAM), не выделенный VM: портал `vibecode02.bitrix24.ru` работает в `resolvedDeployMode: "galaxies-only"`, подтверждено `GET /v1/me` 2026-07-03 (детали контракта деплоя — `specification.md` раздел 2).
- Тесты: Vitest + React Testing Library (фронтенд), Vitest + supertest (бэкенд).
- Монорепо: npm workspaces — `apps/frontend`, `apps/backend`, `packages/shared-types`.
- Node.js установлен на машине пользователя (v24.18.0, ставили через `winget install OpenJS.NodeJS.LTS --source winget`, т.к. напрямую с nodejs.org скачать не получилось — исходящий интернет в песочнице bash/powershell инструментов заблокирован для произвольных сайтов, но `npm install` через реестр npm работает).
- **Важно**: `npm install` уже был проблемный момент — create-vite подтянул Vite 8 + `@vitejs/plugin-react@6` (нужен vite ^8), а `vitest@2.x` требовал vite ^5 → конфликт версий, дублирование vite в node_modules, ошибка типов в `vite.config.ts`. Исправлено апгрейдом `vitest` до `^4.0.0` во всех трёх workspace (поддерживает vite ^8). Если при следующей `npm install` вылезет похожая ошибка типов в `vite.config.ts` (`No overload matches this call`) — это тот же класс проблемы, решение то же (выровнять мажорные версии vite/vitest).

## Известные открытые архитектурные решения (не потерять)

- Полное дерево решений и найденных проблем по каждому пункту макета — в комментариях внутри `design/maket-6.html` (шапка файла, плюс шапки макетов 1–5 для истории) и в `design/handoff.md` (компоненты Ant Design, которые должны заменить самописный JS: `Radio.Group`/`Segmented` вместо ручных `role=radiogroup`, `Modal` вместо самодельной модалки, `Upload.Dragger`, `ColorPicker`).
- PDF-экспорт бланка заказа (кнопка «Скачать файл заказа в PDF») в макете сейчас — client-side демо на jsPDF (CDN), это ВРЕМЕННО и не является финальной архитектурой. Реальный экспорт — бэкенд + Puppeteer/HTML-шаблон (или fallback без headless-браузера, см. пункт про Puppeteer выше). Пользователь явно попросил не трогать эту часть макета до реального внедрения на портал. Отдельно: AVIF→PDF пайплайн сжатия макетов (раздел 4.1) — тоже пока клиентская имитация (setTimeout, jsPDF-заглушка с реальными байтами для превью), реальное AVIF-кодирование (`@jsquash/avif` — кандидат) — Этап 2.
- Зона печати "Спина" и вид сзади/спереди в SVG-превью — реализованы только для типа "худи" (как и вся остальная SVG-иллюстрация); для футболки/свитшота/лонгслива превью показывает заглушку-подпись "шаблон появится позже" — это осознанное ограничение MVP, зафиксировано в спецификации, реальные шаблоны — Этап 2.
