# Конструктор заказов одежды с принтом (интеграция с Битрикс24)

## Цель проекта

Разработать MVP веб-конструктора для оформления заказов на печать одежды (футболки, худи, свитшоты и т.д.) с возможностью загрузки макета, выбора параметров изделия (тип, ткань, цвет, размеры) и комментария. Приложение должно быть встроено в карточку сделки Битрикс24 через виджет (iframe) и обмениваться данными через REST API Битрикс24, создавая или обновляя сделку с заполненными пользовательскими полями и прикреплёнными файлами макетов. 

## Основные характеристики MVP

-  Должна быть возможность по результатам заполнения всех полей скачивать PDF или JPEG файл.
- **Интерактивный конструктор** с визуальным SVG-превью изделия, обновляемым в реальном времени.
- **Загрузка файлов** (макет принта) с поддержкой Drag&Drop.
- **Выбор параметров**:
  - Тип изделия (футболка, худи, свитшот, лонгслив)
  - Ткань (футер 2х или 3х нитка, с начесом или без)
  - Уход за материалом (долговечный — стирка 60° / бережный уход — стирка 40°)
  - Способ нанесения принта (ДТФ / сублимация / вышивка)
  - Цвет (палитра, 2 зоны — основной цвет и цвет отделки)
  - Размер и количество (таблица от S до 7XL)
  - Дополнительные опции (капюшон, карман, молния, отделка и др.)
  - Зона размещения принта (грудь слева / вся грудь / рукав) — на SVG-превью загруженный макет отображается в выбранной зоне
  - Текстовый комментарий

(Уход за материалом, способ нанесения принта и зона размещения принта — добавлены по итогам согласования Итерации 1 чернового дизайна, 2026-07-01; исходный перечень см. в истории или `specification.md`, раздел 3.)
- **Интеграция с Битрикс24**:
  - Виджет внутри карточки сделки.
  - При отправке формы создаётся новая сделка (или обновляется существующая) с заполненными пользовательскими полями.
  - Загруженный файл прикрепляется к сделке через Диск Битрикс24.
- **Хранение данных**: на старте данные передаются напрямую в Битрикс24, локальное хранение не предусмотрено (опционально – логирование для отладки).

## Архитектура проекта

Решение состоит из трёх основных компонентов:
- Фронтенд (React SPA) – пользовательский интерфейс конструктора.

- Бэкенд (Node.js + Express) – API-прослойка для обработки запросов и взаимодействия с Битрикс24.-

- Битрикс24 – целевая CRM-система, в которую передаются данные.

### Схема взаимодействия

1. Пользователь открывает виджет в карточке сделки Битрикс24.
2. Фронтенд получает параметр `deal_id` из URL.
3. Пользователь заполняет форму и загружает файл.
4. Фронтенд отправляет POST-запрос на бэкенд `/api/order` с данными (multipart/form-data, включая файл).
5. Бэкенд валидирует данные, авторизуется в Битрикс24 через OAuth 2.0 и выполняет:
   - Если `deal_id` передан – обновляет сделку (`crm.deal.update`).
   - Если `deal_id` отсутствует – создаёт новую сделку (`crm.deal.add`).
   - Загружает файл в Диск Битрикс24 (`disk.folder.uploadfile`) и привязывает к сделке (через поле или комментарий).
6. Возвращает фронтенду статус операции.

### Детали реализации

- **Авторизация**: OAuth 2.0 с автоматическим обновлением `refresh_token` (используется официальный SDK `B24JsSDK`).
- **Пользовательские поля**: в Битрикс24 создаются поля для хранения выбранных параметров (тип, ткань, цвет, размеры, опции). Размеры и количества сохраняются в виде JSON-строки в текстовом поле.
- **Файлы**: принимаются форматы .png, .jpg, .pdf, .ai, .eps. Максимальный размер – 115 МБ.

## Технологический стек

| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Фронтенд** | React (с хуками), Redux Toolkit (для состояния), TypeScript, сборка на **Vite** | Современный, компонентный подход, удобное управление сложным состоянием формы и SVG-превью; Vite быстрее CRA в dev/HMR и проще для встраивания в iframe. |
| **UI-библиотека** | **Ant Design** (финальный выбор) | Готовые компоненты (селекты, загрузчики файлов, палитра) ускоряют разработку. |
| **SVG-превью** | React + JSX (SVG) | Динамическое обновление через изменение атрибутов и видимости элементов. |
| **Бэкенд** | Node.js + Express.js + TypeScript | Асинхронный, высокопроизводительный, единый язык с фронтендом, размещается на инфраструктуре BlackHole VibeCode. |
| **Тестирование** | Vitest + React Testing Library (фронтенд), Vitest + supertest (бэкенд) | Единый test-runner на весь монорепозиторий, минимум конфигурации. |
| **Монорепозиторий** | npm workspaces: `apps/frontend`, `apps/backend`, `packages/shared-types` | Расшаривание TS-типов между фронтендом и бэкендом без лишней инфраструктуры. |
| **Хранилище файлов** | Диск Битрикс24 (облачное), доступ через VibeCode Gateway | Минимизирует нагрузку на сервер, файлы сразу привязаны к сделке. |
| **Окружение** | Node.js 18+, npm 8+, dotenv | Для управления переменными окружения. Секреты (`vibe_app_...`, `vibe_api_...`) — только в `.env`, не в репозитории. |

Примечание: проект строится на платформе Битрикс24 **VibeCode**, а не на классическом REST-приложении с ручным OAuth 2.0 — см. `specification.md`, разделы 2 и 2.1, для актуальной архитектуры, порядка итераций (дизайн → дизайн-полировка → каркас кода) и открытых технических рисков.

## План работ (этапы разработки)

Актуальный порядок работ и этапы — см. `specification.md`, разделы 2.1 и 8 (этот раздел ниже сохранён как исходная оценка по дням; актуализирован состав этапов под платформу VibeCode и добавлены дизайн-итерации перед стартом кодирования).

### Этап 0: Подготовка инфраструктуры (1 день)
- Настройка локального сервера (Express).
- Регистрация приложения в Битрикс24 (получение client_id, client_secret).
- Создание пользовательских полей в сделке (типы: строка, текст, файл).
- Определение структуры JSON для размера/количества.

### Этап 1: Бэкенд – базовое API и интеграция с Битрикс24 (2–3 дня)
- Реализовать эндпоинт `POST /api/order` с валидацией.
- Реализовать OAuth-авторизацию и хранение токенов.
- Написать функции:
  - `createOrUpdateDeal(dealData, dealId?)` – создание/обновление сделки с заполнением пользовательских полей.
  - `uploadFileToDisk(file, dealId)` – загрузка файла в Диск и прикрепление к сделке.
- Обработка ошибок и логирование.

### Этап 2: Фронтенд – конструктор и SVG-превью (3–4 дня)
- Разработать компоненты:
  - Загрузчик файлов (Drag&Drop).
  - Выбор типа изделия (карточки).
  - Выбор ткани, цвета, опций (чекбоксы/переключатели).
  - Таблица размеров с полями ввода количества.
  - Комментарий.
- Реализовать динамическое SVG-превью:
  - Подготовить базовые SVG-шаблоны для каждого типа изделия.
  - При изменении параметров обновлять SVG (цвет, видимость элементов).
- Собрать данные в объект и отправить на бэкенд.
- Обработать состояния загрузки, успеха/ошибок.

### Этап 3: Создание виджета в Битрикс24 (1 день)
- Зарегистрировать виджет (в разделе «Виджеты» или «Приложения»).
- Указать URL вида `https://ваш-домен/constructor?deal_id=#ID#`.
- Настроить права доступа для виджета.
- Встроить виджет в карточку сделки (вкладка или сайдбар).

### Этап 4: Тестирование и отладка (2 дня)
- Проверить все сценарии: создание новой сделки, обновление существующей, загрузка разных типов файлов.
- Проверить работу OAuth (обновление токенов).
- Проверить корректную работу внутри iframe (CORS, адаптивность).
- Исправить выявленные ошибки.

### Этап 5: Деплой и документация (1–2 дня)
- Подготовить production-сборку фронтенда.
- Настроить сервер (локальный или облачный).
- Написать инструкцию по развертыванию и настройке (см. раздел «Установка»).
- Создать итоговый README.md.

**Итого**: примерно 10–12 рабочих дней.
