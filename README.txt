## Цель проекта

Разработать MVP веб-конструктора для оформления заказов на печать одежды (футболки, худи, свитшоты и т.д.) с возможностью загрузки макета, выбора параметров изделия (тип, ткань, цвет, размеры) и комментария. Приложение должно быть встроено в карточку сделки Битрикс24 через виджет (iframe) и обмениваться данными через REST API Битрикс24, создавая или обновляя сделку с заполненными пользовательскими полями и прикреплёнными файлами макетов. 

## Основные характеристики MVP

-  Должна быть возможность по результатам заполнения всех полей скачивать PDF или JPEG файл.
- **Интерактивный конструктор** с визуальным SVG-превью изделия, обновляемым в реальном времени.
- **Загрузка файлов** (макет принта) с поддержкой Drag&Drop.
- **Выбор параметров**:
  - Тип изделия (футболка, худи, свитшот, лонгслив)
  - Ткань (футер 2х или 3х нитка, с начесом или без)
  - Уход за материалом (долговечный — стирка 60° / бережный уход — стирка 40°)
  - Способ нанесения принта (ДТФ / сублимация / вышивка)
  - Цвет (палитра, 2 зоны — основной цвет и цвет отделки)
  - Размер и количество (таблица от S до 7XL)
  - Дополнительные опции (капюшон, карман, молния, отделка и др.)
  - Зона размещения принта (грудь слева / вся грудь / рукав) — на SVG-превью загруженный макет отображается в выбранной зоне
  - Текстовый комментарий

(Уход за материалом, способ нанесения принта и зона размещения принта — добавлены по итогам согласования Итерации 1 чернового дизайна, 2026-07-01; исходный перечень см. в истории или `specification.md`, раздел 3.)
- **Интеграция с Битрикс24**:
  - Виджет внутри карточки сделки.
  - При отправке формы создаётся новая сделка (или обновляется существующая) с заполненными пользовательскими полями.
  - Загруженный файл прикрепляется к сделке через Диск Битрикс24.
- **Хранение данных**: на старте данные передаются напрямую в Битрикс24, локальное хранение не предусмотрено (опционально – логирование для отладки).

### Схема взаимодействия (актуальная — платформа VibeCode, не классический REST+OAuth2)

1. Пользователь открывает вкладку в карточке сделки (placement `CRM_DEAL_DETAIL_TAB`).
2. Gateway передаёт контекст сделки как `placement_options` (JSON-строка в query-параметрах), а не как `deal_id` напрямую — фронтенд парсит его сам.
3. Пользователь заполняет форму и загружает файл.
4. Фронтенд отправляет POST на бэкенд `/api/order` (`multipart/form-data`).
5. Бэкенд определяет пользователя через заголовок `X-Vibe-Authorization` (Gateway подставляет его сам) → `GET /v1/me`, затем:
   - `dealId` передан — `PATCH /v1/deals/{id}`; иначе — `POST /v1/deals`;
   - загружает файл на Диск (`POST /v1/files/upload`, до 52 МБ) и привязывает к сделке.
6. Возвращает статус операции фронтенду.

Полная архитектура, форматы ответов и найденные на практике нюансы API — `specification.md`, разделы 2 и 7.

### Детали реализации

- **Авторизация**: ключи VibeCode (`vibe_api_...` личный / `vibe_app_...` для встраивания), без ручного OAuth 2.0 — подробности `specification.md`, раздел 2.
- **Пользовательские поля**: создаются программно через API Битрикс24 (`POST /v1/userfields/deals`), хранят параметры заказа; размеры и опции — JSON-строкой.
- **Файлы**: .png, .jpg, .pdf, .ai, .eps, максимум 52 МБ (лимит Диска Битрикс24 через VibeCode — расчёт в `specification.md`, раздел 4). Растровые `.png`/`.jpg` больше лимита — клиентское адаптивное сжатие в AVIF (потолок 48 МБ) с упаковкой в PDF (потолок 52 МБ, итоговый производственный файл) вместо отказа; подробности — раздел 4.1, дизайн — `design/maket-6.html`.

## Технологический стек

| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| **Фронтенд** | React (с хуками), Redux Toolkit (для состояния), TypeScript, сборка на **Vite** | Современный, компонентный подход, удобное управление сложным состоянием формы и SVG-превью; Vite быстрее CRA в dev/HMR и проще для встраивания в iframe. |
| **UI-библиотека** | **Ant Design** (финальный выбор) | Готовые компоненты (селекты, загрузчики файлов, палитра) ускоряют разработку. |
| **SVG-превью** | React + JSX (SVG) | Динамическое обновление через изменение атрибутов и видимости элементов. |
| **Бэкенд** | Node.js + Express.js + TypeScript | Асинхронный, высокопроизводительный, единый язык с фронтендом, размещается на инфраструктуре BlackHole VibeCode. |
| **Тестирование** | Vitest + React Testing Library (фронтенд), Vitest + supertest (бэкенд) | Единый test-runner на весь монорепозиторий, минимум конфигурации. |
| **Монорепозиторий** | npm workspaces: `apps/frontend`, `apps/backend`, `packages/shared-types` | Расшаривание TS-типов между фронтендом и бэкендом без лишней инфраструктуры. |
| **Хранилище файлов** | Диск Битрикс24 (облачное), доступ через VibeCode Gateway | Минимизирует нагрузку на сервер, файлы сразу привязаны к сделке. |

## Установка и запуск (локально)

### Требования
- Node.js 18+ (проверено на v24.18.0)
- npm 8+ (проверено на 11.16.0)
- Тестовый портал Битрикс24 с правами администратора

### Структура монорепозитория (npm workspaces)
- `apps/frontend` — React + TypeScript + Vite, Redux Toolkit, Ant Design.
- `apps/backend` — Node.js + Express + TypeScript (BFF).
- `packages/shared-types` — общие TS-типы формы заказа и маппинг на UF_CRM_* поля сделки.

### Команды (из корня репозитория)
- `npm install` — установить зависимости всех workspace сразу.
- `npm run dev:frontend` / `npm run dev:backend` — запустить фронтенд/бэкенд в режиме разработки (фронтенд на `:5173` проксирует `/api` на бэкенд `:3001`, см. `apps/frontend/vite.config.ts`).
- `npm run build` — собрать все workspace (важен порядок: `packages/shared-types` собирается первым, от него зависит `apps/backend`).
- `npm test` — прогнать тесты всех workspace (Vitest).
- `npm run lint` — проверить весь монорепозиторий ESLint.
- `npm run vibecode:check-server` — проверить живой BlackHole URL; полная owner-only проверка требует `VIBE_PERSONAL_API_KEY`.
- `npm run vibecode:setup-portal` — создать UF-поля и привязать placement на актуальном портале после установки правильного `VIBE_APP_KEY`.
- Секреты — скопировать `.env.example` в `.env` и заполнить: `VIBE_APP_KEY` (ключ приложения VibeCode), `VIBE_LAYOUT_FOLDER_ID` (папка на Диске для макетов), `VIBE_API_BASE_URL` (опционально). Файл `.env` не коммитится.
