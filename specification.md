## 1. Цель проекта

MVP веб-конструктора для оформления заказов на печать одежды (футболки, худи, свитшоты, лонгсливы) со следующими возможностями:

- визуальный SVG-конструктор изделия с превью в реальном времени;
- загрузка макета принта (Drag&Drop);
- выбор параметров изделия и печать сводного PDF/JPEG документа;
- встраивание в карточку сделки Битрикс24, автоматическое создание/обновление сделки и прикрепление файла макета.

## 2. Архитектура

Проект строится на платформе **Битрикс24 VibeCode** — это меняет архитектуру относительно классического REST-приложения:

| Компонент | Решение |
|---|---|
| Frontend | React SPA (TypeScript, хуки, Redux Toolkit), сборка на **Vite** (вместо CRA — быстрее dev/HMR, проще настройка base path и dev-proxy к BFF при встраивании в iframe) |
| UI-библиотека | **Ant Design** (финальный выбор) |
| SVG-превью | React + JSX/SVG, модульные перекрашиваемые группы |
| Backend (BFF) | Node.js + Express + TypeScript, размещается на инфраструктуре BlackHole (Битрикс24), а не на своём хостинге |
| Тестирование | **Vitest + React Testing Library** (frontend), **Vitest + supertest** (backend) — единый test-runner на весь монорепозиторий |
| Монорепозиторий | **npm workspaces**: `apps/frontend`, `apps/backend`, `packages/shared-types` (общие TS-типы формы заказа и маппинга UF_CRM_*) |
| Линтинг/форматирование | ESLint + Prettier, единый корневой конфиг с overrides для frontend/backend |
| Авторизация | Ключ `vibe_app_...` (скоупы: `crm`, `disk`), без ручного OAuth 2.0 — сессия и идентификация пользователя приходят через Gateway (заголовок `X-Vibe-Authorization` → `GET /v1/me`). Детали двух типов ключей и формата ответа — ниже, живая проверка 2026-07-03 |
| Работа с CRM | `POST /v1/deals`, `PATCH /v1/deals/{id}` — создание/обновление сделки с полями `UF_CRM_*` |
| Файлы | Диск Битрикс24: `GET /v1/storages` → `folderId` → `POST /v1/files/upload` (base64 JSON, до 52 МБ) → привязка к сделке |
| Встраивание | Placement `CRM_DEAL_DETAIL_TAB`, привязка через `POST /v1/placements/bind` (только ключом `vibe_app_...`); контекст сделки приходит в query-параметре `placement_options` (JSON-строка `{"ID":"..."}`, не отдельным `deal_id`) |
| Хранилище данных | Собственная БД не используется — состояние живёт в сделке Битрикс24 + файле на Диске |
| Деплой | `POST /v1/infra/servers/:id/deploy` — приложение доступно по адресу вида `https://app-xxx.vibecode.bitrix24.tech` |

**Предварительное условие:** на портале Битрикс24 должна быть активна подписка **BitrixGPT + Маркетплейс** — без неё платформа VibeCode недоступна.

**Разведка живой документации и API завершена 2026-07-03** (тестовый ключ `vibe_api_...`, портал `vibecode02.bitrix24.ru`) — часть страниц документации оказалась неактуальной (например, `/docs-content/recipes/disk-files.md` утверждает, что загрузка файлов не покрыта Entity API — это устарело, метод реально работает, проверено живым запросом с последующим удалением тестового файла). Итоговые ответы:

- **Авторизация**: два разных ключа для двух разных сценариев.
  - `vibe_api_...` (личный ключ) — заголовок `X-Api-Key: vibe_api_...`, без `Bearer`. Годится для фоновых операций и серверных вызовов от имени приложения, но **не может** привязать placement и не участвует в прозрачной iframe-авторизации.
  - `vibe_app_...` (ключ авторизации/OAuth-приложения, раздел «Ключи авторизации» на портале) — обязателен для встраивания вкладки в карточку сделки. При открытии вкладки Gateway обменивает одноразовый `__init`-JWT на сессию и сам подставляет заголовок `X-Vibe-Authorization: Bearer vibe_session_...` на все последующие запросы браузера к бэкенду приложения (токен не виден на клиенте). Бэкенд просто пробрасывает этот заголовок как `Authorization` в `GET /v1/me`, чтобы получить личность пользователя — здесь исходное предположение спецификации о `X-Vibe-Authorization` подтвердилось, но только для этого сценария и только с `vibe_app_` ключом.
  - Формат ответа `GET /v1/me` — единый конверт `{success, data:{...}}`, для личного ключа `data.type:"personal"`, для OAuth-приложения `data.type:"oauth_app"`; в обоих случаях `data.scopes[]`, `data.auth.header`, `data.accessMode`, `data.expiresAt`. Ошибки — `{success:false, error:{code, message}}`, коды: `INVALID_API_KEY`/`KEY_EXPIRED`/`KEY_INACTIVE`/`TOKEN_MISSING` (401), `IP_NOT_ALLOWED` (403), `RATE_LIMITED` (429, с заголовком `Retry-After`).
- **Загрузка файла на Диск**: `POST /v1/files/upload`, JSON-тело `{folderId|storageId, filename, content}` (`content` — base64, **не** multipart), `Content-Type: application/json`, требует scope `disk`. Лимит тела запроса — 70 МБ, из-за раздувания base64 (+33%) реальный максимум исходного файла — **52 МБ** (расчёт и обоснование лимита — раздел 4). Потоковой/chunked-загрузки в этом методе нет — бэкенд обязан полностью прочитать файл и закодировать в base64 перед вызовом.
- **Scope Диска**: `disk` (не отдельный вибкод-скоуп; `vibe:storage` — это отдельное файловое хранилище платформы, не связанное с Диском Битрикс24 и не подходящее для вложений в сделку).
- **Puppeteer/headless Chromium**: подтверждения поддержки в документации не найдено (проверены `/docs/infra`, `/docs/infra/app-runtime` — ни слова про Chromium/Puppeteer/Playwright/sandbox). Косвенный риск-сигнал: деплой на этом портале идёт только в режиме `galaxyApp` (общий хост, контейнер), и в документации по восстановлению после `GALAXY_APP_START_FAILED` упоминается OOM-лимит контейнера ~512 МБ — тесно для headless Chromium вместе с Node-процессом. Решение по-прежнему принимается эмпирической проверкой в начале Этапа 1 (задеплоить мини-пример с Puppeteer), fallback на `pdf-lib`/`@react-pdf/renderer` остаётся в силе и теперь выглядит более вероятным исходом.
- **`POST /v1/infra/servers/:id/deploy`**: портал работает в режиме `resolvedDeployMode: "galaxies-only"` — деплой только как `galaxyApp` (общий хост), не выделенный VM (`standalone`, недоступен на этом портале). Dockerfile не нужен — генерируется автоматически из `runtime`/`install`/`start`/`port`. Обязательные поля: `runtime` (например `node20`), `start` (одна команда). `source.content` — **только** base64-архив (`source.url`/`source.versionId` запрещены для galaxyApp, `400 GALAXY_DEPLOY_CONTENT_ONLY`). `install` — обязательно одна строка без переноса. Постоянные данные — только в `/data` (переживает передеплой, всё остальное стирается). Переменные окружения передаются через `env`. Лимиты: exec timeout 600 сек, конкурентность инлайн-деплоя — 2 на весь бэкенд платформы (`429 DEPLOY_BACKEND_BUSY` при превышении). Перед деплоем нужно вписать `<link rel="icon">` на `/api/server-icons/:id` в HTML и отдельно залить SVG-иконку через `POST /v1/infra/servers/:id/icon`.
- **`POST /v1/placements/bind`**: требует `vibe_app_` ключ (placement scope + Bearer-сессия) — личным ключом недоступно, подтверждено `GET /v1/me` для личного ключа. `CRM_DEAL_DETAIL_TAB` подтверждён живым запросом `GET /v1/placements/available` как валидный код (раздел «CRM Detail Tabs», всего 55 доступных placement-кодов). Получение `deal_id` устроено **не так**, как предполагала предыдущая версия спецификации: `POST /v1/bitrix-handler` обменивает код и делает 302-редирект iframe на `<appUrl>/?placement=...&member_id=...&placement_options=<JSON>&__init=<JWT>`; для `*_DETAIL_TAB` контекст сущности приходит в `placement_options` — JSON-строке вида `{"ID":"42"}` в query-параметрах, а не как отдельный `deal_id`. Парсить через `URLSearchParams.get('placement_options')` + `JSON.parse`, **не** через `BX24.placement.info()` (SDK не видит контекст через handler-редирект). Для self-hosted `appUrl` (не поддомен BlackHole) схема другая — код обменивается вручную через `POST /v1/oauth/token`.
- **Создание UF-полей**: неожиданный бонус — поля можно создавать программно через `POST /v1/userfields/deals` (`{userTypeId, fieldName, label}`), без ручного захода в админку портала. Каталог типов — `GET /v1/userfields/deals/types`, включает нужный нам `file` (для поля «Файл макета» напрямую на сделке — тип-кандидат для эксперимента на Этапе 1, вместо отдельного текстового поля со ссылкой на Диск). На момент разведки на портале 0 пользовательских полей у сделок (`GET /v1/userfields/deals` → `[]`) — все 11 полей из раздела 6 предстоит создать с нуля.

**Секреты**: ключи вида `vibe_app_...` / `vibe_api_...` хранятся только в `.env` (не коммитится) или в секретах платформы VibeCode — никогда в спецификации, README, коде или истории коммитов. Ключ `vibe_api_iPaRzax8QCws4aoZ3HTYwIusVxuu2fej_c3033f`, использованный для разведки 2026-07-03, — тестовый, персональный, не даёт доступа к встраиванию (см. выше); для Этапа 1 понадобится дополнительно ключ `vibe_app_...`.

## 3. Параметры конструктора (объём — строго по исходному README)

### 3.1 Тип изделия
Футболка, худи, свитшот, лонгслив.

### 3.2 Ткань
Футер 2х или 3х нитка, с начёсом или без.

### 3.3 Цвет — 2 зоны
Принято решение об усреднённом варианте детализации:

- **Основной цвет** — цвет тела/силуэта изделия;
- **Цвет отделки** — цвет элементов отделки (капюшон, манжеты, низ, карман, шнурки — то есть те детали, которые в README перечислены в "дополнительных опциях").

SVG-превью строится из **модульных перекрашиваемых групп** (минимум: тело/силуэт как одна группа с основным цветом, элементы отделки как вторая группа с цветом отделки), а не единой заливки на всё изделие.

### 3.4 Размер и количество
Таблица S–7XL с полями ввода количества по каждому размеру.

### 3.5 Дополнительные опции
Капюшон, карман, молния, отделка и др. (реализуются как чекбоксы/переключатели, влияющие на видимость соответствующих элементов SVG).

### 3.6 Комментарий
Свободный текст.

### 3.7 Уход за материалом
Одиночный выбор из двух вариантов: **Долговечный** (стирка 60°) / **Бережный уход** (стирка 40°). Влияет только на данные заказа (UF-поле), на SVG-превью не отображается.

### 3.8 Способ нанесения принта
Одиночный выбор: **ДТФ**, **Сублимация**, **Вышивка**.

### 3.9 Зона размещения принта
Одиночный выбор из трёх зон на изделии: **Грудь слева**, **Вся грудь**, **Рукав**. На SVG-превью выбранная зона отмечена пунктирной рамкой; после загрузки растрового макета (`.png`/`.jpg`) файл отображается внутри рамки, при смене зоны — переносится без повторной загрузки. Для векторных/PDF-макетов вместо превью — плашка с расширением файла (см. раздел 4). Размещение на спине — вне MVP, требует отдельного вида изделия сзади; рассматривается как расширение после Итерации 3 (см. раздел 8, Этап 2).

> **Решение по расширению функциональности:** поля "Тираж", "Сигнальный образец", "Бирки", "Силуэт", замеченные в референсном проекте (`constructor.all4promo.ru`), в MVP **не включаются** — остаёмся в границах исходного README. Параметры 3.7–3.9 (уход за материалом, способ нанесения принта, зона размещения принта), напротив, добавлены по итогам согласования Итерации 1 — 2026-07-01 — и входят в объём MVP.

## 4. Загрузка файлов макета

- Форматы: `.png`, `.jpg`, `.pdf`, `.ai`, `.eps`.
- **Максимальный размер: 52 МБ** (скорректировано 2026-07-03 по факту разведки живого API, было 115 МБ в исходном ТЗ — см. раздел 2). Причина: `POST /v1/files/upload` на Диск Битрикс24 принимает файл только как base64 в JSON-теле, лимит тела запроса на платформе — 70 МБ; base64 раздувает размер в 4/3 раза (+33%), поэтому исходный файл не должен превышать `70 × 0.75 = 52.5` МБ — 52 МБ взято с небольшим запасом на служебные JSON-поля. Проверка исходного размера — на фронтенде и на бэкенде, до base64-кодирования, чтобы не тратить время на заведомо неудачную загрузку (`422 BITRIX_ERROR` при превышении на стороне Битрикс24).
- Загрузка на бэкенд от фронтенда — потоком (streaming, `multipart/form-data`), не в память целиком; но перед финальной отправкой на Диск Битрикс24 файл неизбежно буферизуется и кодируется в base64 целиком — метод API не поддерживает потоковую/chunked-загрузку (см. раздел 2). При 52 МБ максимума это приемлемо.
- Drag&Drop-зона.
- Для `.ai`/`.eps` — живой предпросмотр в браузере невозможен (векторные форматы Illustrator/EPS); показываем иконку-заглушку с именем файла. Сам файл при этом полноценно загружается на Диск Битрикс24 и прикрепляется к сделке.
- Валидация MIME по содержимому файла, а не только по расширению.

### 4.1 Обработка объёмных растровых файлов (AVIF-сжатие)

Согласовано с пользователем 2026-07-03. Проблема: файлы вроде 8K UHD (7680×4320) весят в среднем 126–150 МБ — заведомо больше лимита в 52 МБ (раздел выше). Вместо простого отказа — предлагаем пользователю сжатие.

- **Область действия — только растровые форматы `.png`/`.jpg`/`.jpeg`.** Векторные `.pdf`/`.ai`/`.eps` не поддерживают AVIF-сжатие в принципе (AVIF — растровый кодек). **Рассмотрено и сознательно отложено 2026-07-03**: сжатие объёмного PDF технически возможно двумя путями — (а) растеризация страниц через `pdf.js` на клиенте с последующим прогоном через тот же пайплайн AVIF→PDF, ценой потери векторной точности текста/линий (нежелательно для вышивки и части ДТФ/сублимации крупного формата); (б) настоящее PDF-сжатие с сохранением вектора (по типу Ghostscript) — требует бэкенда, так как клиентского/WebAssembly аналога нет, что противоречит принципу «крупный файл не должен долетать до и так ограниченного `galaxyApp`-бэкенда». Пользователь выбрал не делать ни то, ни другое пока — для `.pdf`/`.ai`/`.eps` больше 52 МБ остаётся только информационное сообщение без сжатия (см. ниже). Возвращаться к вопросу только по отдельному запросу пользователя.
- **Проверка исходного размера — на клиенте, сразу после выбора/drop файла, до какой-либо отправки на бэкенд.** Ветвление по размеру и формату:
  - Файл ≤ 52 МБ (любой формат) → обычный сценарий, без модалки, без изменений.
  - Файл > 52 МБ, формат `.png`/`.jpg`/`.jpeg` → модалка с двумя кнопками: **«Сжать файл»** / **«Выбрать другой»**.
  - Файл > 52 МБ, формат `.pdf`/`.ai`/`.eps` → **без кнопки сжатия**, информационное сообщение: *«Сжатие доступно только для файлов формата .png и .jpg. Ваш файл слишком большой — сконвертируйте его в подходящий формат или выберите другой файл»* (текст — черновой, финальная формулировка — на прогоне `design:ux-copy` при дизайн-итерации для этой модалки).
- **«Выбрать другой»** — закрывает модалку, сбрасывает дропзону, как обычное удаление файла.
- **«Сжать файл» — двухэтапный пайплайн, уточнён 2026-07-03 (второй раунд правок в тот же день):**
  1. **Адаптивное AVIF-кодирование на клиенте, в браузере, через WebAssembly** (кандидат-библиотека: `@jsquash/avif`, на основе libavif; финальный выбор — на Этапе 2). «Адаптивное» — степень сжатия (качество, при необходимости — разрешение как вторая ступень) подбирается автоматически под конкретный файл итеративным поиском, а не фиксированным коэффициентом: цель — максимальное качество, укладывающееся в потолок **48 МБ**. Если даже минимально приемлемое качество не укладывается в 48 МБ (редкий edge case, экстремальное разрешение) — следующий рычаг понижения размера — уменьшение пиксельных размеров изображения.
  2. **Конвертация сжатого AVIF в PDF** (одностраничный документ с встроенным изображением). Причина отдельного шага: стандартные PDF-библиотеки не умеют встраивать AVIF-поток напрямую — растровые данные из AVIF декодируются и переупаковываются в поддерживаемый PDF растровый формат (JPEG/PNG внутри PDF), отсюда и запас между потолками (48 МБ AVIF → 52 МБ итоговый PDF, ~4 МБ на накладные расходы переупаковки). Итоговый PDF — **не более 52 МБ**.
  - Причины именно клиентской обработки (оба шага), а не серверной: файл весом до 150 МБ не должен долетать до бэкенда вовсе, если пользователь может обработать его локально — меньше нагрузки на и так ограниченный `galaxyApp`-контейнер (~512 МБ RAM, раздел 2), который вдобавок должен тянуть Puppeteer; согласуется с принципом «большой файл не должен буферизоваться на сервере целиком без необходимости».
- **Итоговый PDF — окончательный производственный файл**, согласованное решение пользователя: именно этот PDF (не промежуточный AVIF) — то, что идёт в SVG-превью, что автоматически скачивается пользователю на компьютер (`<a download>`/blob-URL, до конвертации в AVIF/PDF ничего не скачивается) и что загружается на Диск Битрикс24 и прикрепляется к сделке. Переход на PDF как финальный формат (вместо AVIF в первой версии этого решения) снимает риск несовместимости с печатным оборудованием/RIP-софтом — PDF читают повсеместно, в отличие от AVIF.
  - **Живое превью PDF на изделии — решено 2026-07-03 (третий раунд правок в тот же день)**: пользователь попросил показывать реальное содержимое PDF в зоне печати, а не декодированный AVIF-битмап отдельно от файла. Реализовано через `pdf.js` — страница PDF рендерится в канвас прямо в браузере (`getDocument` → `getPage(1)` → `page.render` в `<canvas>` → `canvas.toDataURL()`), результат вставляется в SVG `<image>` тем же способом, что и `.png`/`.jpg`. Работает как для PDF, загруженного пользователем напрямую (≤52 МБ, без сжатия), так и для итогового PDF после AVIF-сжатия — технически это один и тот же файл и в превью, и в том, что уходит на Диск/скачивается (расхождение между превью и итоговым файлом, обсуждавшееся раньше, снято). Для `.ai`/`.eps` живого превью по-прежнему нет — нет клиентского способа их отрендерить, остаётся плашка-заглушка с расширением.
- Если после сжатия AVIF (48 МБ) итоговый PDF всё же вышел бы за 52 МБ (маловероятно, но возможный edge case из-за переупаковки) — поведение уточняется на Этапе 2 (кандидат по умолчанию: более агрессивное качество AVIF и повтор конвертации; при повторной неудаче — тот же сценарий, что и для векторных форматов выше).
- **Макет модалки спроектирован и подтверждён пользователем как финальный** — [`design/maket-6.html`](design/maket-6.html) (поверх согласованного макета №5), включает демо-кнопки для прогона обоих сценариев (растровый/векторный) без необходимости реального крупного файла.

## 7. Встраивание в Битрикс24

1. Пользователь открывает вкладку в карточке сделки (placement `CRM_DEAL_DETAIL_TAB`, подтверждён живым `GET /v1/placements/available`). Требует ключ `vibe_app_...`, привязку делает `POST /v1/placements/bind` (личным ключом `vibe_api_...` недоступно — см. раздел 2).
2. Gateway обменивает одноразовый `__init`-JWT на сессию и редиректит iframe на `<appUrl>/?placement=CRM_DEAL_DETAIL_TAB&member_id=...&placement_options=<JSON>&__init=...`. Frontend **не** получает `deal_id` напрямую — вместо этого парсит query-параметр `placement_options` (JSON-строка вида `{"ID":"42"}`, `URLSearchParams.get('placement_options')` + `JSON.parse`) и достаёт `ID` сделки оттуда. Через `BX24.placement.info()` контекст недоступен (SDK не видит его через handler-редирект).
3. Пользователь заполняет форму, загружает макет.
4. Frontend отправляет данные на BFF (`multipart/form-data`).
5. BFF проверяет идентичность пользователя: Gateway сам подставляет заголовок `X-Vibe-Authorization: Bearer vibe_session_...` на каждый запрос браузера к бэкенду — BFF пробрасывает его как `Authorization` в `GET /v1/me`, валидирует данные:
   - если `dealId` (из `placement_options`) передан — `PATCH /v1/deals/{id}`;
   - если отсутствует (открытие вне карточки сделки, если такой сценарий разрешён) — `POST /v1/deals`;
   - загружает файл на Диск (`POST /v1/files/upload`, до 52 МБ — раздел 4) и привязывает к сделке.
6. Возвращает статус операции на frontend.
