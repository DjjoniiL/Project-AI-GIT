## 1. Цель проекта

MVP веб-конструктора для оформления заказов на печать одежды (футболки, худи, свитшоты, лонгсливы) со следующими возможностями:

- визуальный SVG-конструктор изделия с превью в реальном времени;
- загрузка макета принта (Drag&Drop);
- выбор параметров изделия и печать сводного PDF/JPEG документа;
- встраивание в карточку сделки Битрикс24, автоматическое создание/обновление сделки и прикрепление файла макета.

## 2. Архитектура

Проект строится на платформе **Битрикс24 VibeCode** — это меняет архитектуру относительно классического REST-приложения:

| Компонент | Решение |
|---|---|
| Frontend | React SPA (TypeScript, хуки, Redux Toolkit), сборка на **Vite** (вместо CRA — быстрее dev/HMR, проще настройка base path и dev-proxy к BFF при встраивании в iframe) |
| UI-библиотека | **Ant Design** (финальный выбор) |
| SVG-превью | React + JSX/SVG, модульные перекрашиваемые группы |
| Backend (BFF) | Node.js + Express + TypeScript, размещается на инфраструктуре BlackHole (Битрикс24), а не на своём хостинге |
| Тестирование | **Vitest + React Testing Library** (frontend), **Vitest + supertest** (backend) — единый test-runner на весь монорепозиторий |
| Монорепозиторий | **npm workspaces**: `apps/frontend`, `apps/backend`, `packages/shared-types` (общие TS-типы формы заказа и маппинга UF_CRM_*) |
| Линтинг/форматирование | ESLint + Prettier, единый корневой конфиг с overrides для frontend/backend |
| Авторизация | Ключ `vibe_app_...` (скоупы: `crm`, `disk`), без ручного OAuth 2.0 — сессия и идентификация пользователя приходят через Gateway (заголовок `X-Vibe-Authorization` → `GET /v1/me`). Детали двух типов ключей и формата ответа — ниже, живая проверка 2026-07-03 |
| Работа с CRM | `POST /v1/deals`, `PATCH /v1/deals/{id}` — создание/обновление сделки с полями `UF_CRM_*` |
| Файлы | Диск Битрикс24: `GET /v1/storages` → `folderId` → `POST /v1/files/upload` (base64 JSON, до 52 МБ) → привязка к сделке |
| Встраивание | Placement `CRM_DEAL_DETAIL_TAB`, привязка через `POST /v1/placements/bind` (только ключом `vibe_app_...`); контекст сделки приходит в query-параметре `placement_options` (JSON-строка `{"ID":"..."}`, не отдельным `deal_id`) |
| Хранилище данных | Собственная БД не используется — состояние живёт в сделке Битрикс24 + файле на Диске |
| Деплой | `POST /v1/infra/servers/:id/deploy` — приложение доступно по адресу вида `https://app-xxx.vibecode.bitrix24.tech` |

**Предварительное условие:** на портале Битрикс24 должна быть активна подписка **BitrixGPT + Маркетплейс** — без неё платформа VibeCode недоступна.

**Живой API проверен запросами** (портал `vibecode02.bitrix24.ru`) — часть страниц документации оказалась неактуальной (например, `/docs-content/recipes/disk-files.md` ошибочно утверждает, что загрузка файлов не покрыта Entity API — метод реально работает). Ключевые факты:

- **Два типа ключей**. `vibe_api_...` (личный) — заголовок `X-Api-Key`, без `Bearer`; годится для фоновых операций, но **не может** привязать placement и не участвует в iframe-авторизации. `vibe_app_...` (OAuth-приложение) — обязателен для встраивания вкладки: при открытии Gateway обменивает одноразовый `__init`-JWT на сессию и сам подставляет заголовок `X-Vibe-Authorization: Bearer vibe_session_...` на запросы браузера к бэкенду; бэкенд пробрасывает это как `Authorization` в `GET /v1/me`, чтобы получить личность пользователя.
- **`GET /v1/me`** — конверт `{success, data:{...}}` (`data.type`: `personal`/`oauth_app`, `data.scopes[]`, `data.auth.header`, `data.accessMode`, `data.expiresAt`). Ошибки — `{success:false, error:{code, message}}`: `INVALID_API_KEY`/`KEY_EXPIRED`/`KEY_INACTIVE`/`TOKEN_MISSING` (401), `IP_NOT_ALLOWED` (403), `RATE_LIMITED` (429, заголовок `Retry-After`).
- **Загрузка файла на Диск**: `POST /v1/files/upload`, JSON-тело `{folderId|storageId, filename, content}` (`content` — base64, **не** multipart), scope `disk`. Лимит тела запроса — 70 МБ → максимум исходного файла **52 МБ** (расчёт — раздел 4). Потоковой/chunked-загрузки в методе нет — файл кодируется в base64 целиком перед вызовом. Scope `vibe:storage` — отдельное файловое хранилище платформы, не связано с Диском Битрикс24 и не годится для вложений в сделку.
- **Puppeteer/headless Chromium**: поддержка документацией не подтверждена и не опровергнута (`/docs/infra`, `/docs/infra/app-runtime` не упоминают Chromium/Puppeteer/Playwright). Риск-сигнал: деплой — только `galaxyApp`-контейнер (~512 МБ RAM, из документации по восстановлению после `GALAXY_APP_START_FAILED`) — тесно для headless Chromium вместе с Node-процессом. Решение — эмпирическая проверка при первом деплое, fallback `pdf-lib`/`@react-pdf/renderer` наготове.
- **`POST /v1/infra/servers/:id/deploy`**: портал работает в режиме `resolvedDeployMode: "galaxies-only"` — только `galaxyApp` (общий хост), не выделенный VM. Dockerfile не нужен — генерируется из `runtime`/`install`/`start`/`port`. Обязательно: `runtime` (`node20`), `start` (одна команда). `source.content` — только base64-архив (`source.url`/`source.versionId` запрещены, `400 GALAXY_DEPLOY_CONTENT_ONLY`). `install` — одна строка без переноса. Постоянные данные — только `/data`. Лимиты: exec timeout 600 сек, конкурентность деплоя — 2 на весь бэкенд платформы (`429 DEPLOY_BACKEND_BUSY`). Перед деплоем — `<link rel="icon">` на `/api/server-icons/:id` в HTML + SVG-иконка через `POST /v1/infra/servers/:id/icon`.
- **`POST /v1/placements/bind`**: требует `vibe_app_` ключ (личным недоступно). `CRM_DEAL_DETAIL_TAB` подтверждён живым `GET /v1/placements/available` (из 55 кодов). Контекст сделки приходит **не** как `deal_id` в query, а как `placement_options` — JSON-строка `{"ID":"42"}` после handler-редиректа (`POST /v1/bitrix-handler` → `<appUrl>/?placement=...&member_id=...&placement_options=<JSON>&__init=<JWT>`); парсить через `URLSearchParams` + `JSON.parse`, **не** через `BX24.placement.info()` (SDK не видит контекст через редирект). Для self-hosted `appUrl` — код обменивается вручную через `POST /v1/oauth/token`.
- **UF-поля создаются программно**: `POST /v1/userfields/deals {userTypeId, fieldName, label}`, каталог типов — `GET /v1/userfields/deals/types` (включает `file` — кандидат для поля «Файл макета» напрямую на сделке). На портале пока 0 полей у сделок — все 11 из раздела 6 предстоит создать.

**Секреты**: ключи `vibe_app_...` / `vibe_api_...` — только в `.env` (не коммитится) или в секретах платформы VibeCode, никогда в спецификации/README/коде/истории коммитов.

Футболка, худи, свитшот, лонгслив.

Футер 2х или 3х нитка, с начёсом или без.

### 3.3 Цвет — 2 зоны
Принято решение об усреднённом варианте детализации:

SVG-превью строится из **модульных перекрашиваемых групп** (минимум: тело/силуэт как одна группа с основным цветом, элементы отделки как вторая группа с цветом отделки), а не единой заливки на всё изделие.

### 3.4 Размер и количество
Таблица S–7XL с полями ввода количества по каждому размеру.

### 3.5 Дополнительные опции
Капюшон, карман, молния, отделка и др. (реализуются как чекбоксы/переключатели, влияющие на видимость соответствующих элементов SVG).

### 3.6 Комментарий
Свободный текст.

### 3.7 Уход за материалом
Одиночный выбор из двух вариантов: **Долговечный** (стирка 60°) / **Бережный уход** (стирка 40°). Влияет только на данные заказа (UF-поле), на SVG-превью не отображается.

### 3.8 Способ нанесения принта
Одиночный выбор: **ДТФ**, **Сублимация**, **Вышивка**.

### 3.9 Зона размещения принта
Одиночный выбор из трёх зон на изделии: **Грудь слева**, **Вся грудь**, **Рукав**. На SVG-превью выбранная зона отмечена пунктирной рамкой; после загрузки растрового макета (`.png`/`.jpg`) файл отображается внутри рамки, при смене зоны — переносится без повторной загрузки. Для векторных/PDF-макетов вместо превью — плашка с расширением файла (см. раздел 4). Размещение на спине — вне MVP, требует отдельного вида изделия сзади; рассматривается как расширение после Итерации 3 (см. раздел 8, Этап 2).

## 4. Загрузка файлов макета

- Форматы: `.png`, `.jpg`, `.pdf`, `.ai`, `.eps`.
- **Максимальный размер: 52 МБ** (скорректировано 2026-07-03 по факту разведки живого API, было 115 МБ в исходном ТЗ — см. раздел 2). Причина: `POST /v1/files/upload` на Диск Битрикс24 принимает файл только как base64 в JSON-теле, лимит тела запроса на платформе — 70 МБ; base64 раздувает размер в 4/3 раза (+33%), поэтому исходный файл не должен превышать `70 × 0.75 = 52.5` МБ — 52 МБ взято с небольшим запасом на служебные JSON-поля. Проверка исходного размера — на фронтенде и на бэкенде, до base64-кодирования, чтобы не тратить время на заведомо неудачную загрузку (`422 BITRIX_ERROR` при превышении на стороне Битрикс24).
- Загрузка на бэкенд от фронтенда — потоком (streaming, `multipart/form-data`), не в память целиком; но перед финальной отправкой на Диск Битрикс24 файл неизбежно буферизуется и кодируется в base64 целиком — метод API не поддерживает потоковую/chunked-загрузку (см. раздел 2). При 52 МБ максимума это приемлемо.
- Drag&Drop-зона.
- Для `.ai`/`.eps` — живой предпросмотр в браузере невозможен (векторные форматы Illustrator/EPS); показываем иконку-заглушку с именем файла. Сам файл при этом полноценно загружается на Диск Битрикс24 и прикрепляется к сделке.
- Валидация MIME по содержимому файла, а не только по расширению.

### 4.1 Обработка объёмных растровых файлов (AVIF → PDF)

Растровые `.png`/`.jpg`/`.jpeg` больше 52 МБ (пример — 8K UHD, 126–150 МБ): вместо отказа — модалка **«Сжать файл»** / **«Выбрать другой»**. Проверка размера — на клиенте, сразу после выбора/drop, до отправки на бэкенд.

Векторные `.pdf`/`.ai`/`.eps` больше лимита — только информационное сообщение, сжатие недоступно. **Рассмотрено и сознательно отложено**: PDF можно было бы сжать через растеризацию `pdf.js` на клиенте (теряет векторную точность текста/линий — нежелательно для вышивки/крупноформатной печати) или настоящим PDF-сжатием с сохранением вектора (требует бэкенда — Ghostscript-класс инструментов, клиентского аналога нет). Не переоткрывать без запроса пользователя.

Пайплайн «Сжать файл»:
1. **Адаптивное AVIF-кодирование на клиенте** (WebAssembly, кандидат `@jsquash/avif`; финальный выбор — Этап 2) — качество (и разрешение как вторая ступень) подбирается итеративным поиском под конкретный файл, цель — максимум качества в пределах потолка **48 МБ**, не фиксированный коэффициент.
2. **Конвертация в PDF** (одностраничный документ со встроенным изображением) — стандартные PDF-библиотеки не embed-ят AVIF-поток напрямую, растровые данные переупаковываются в JPEG/PNG внутри PDF; потолок **52 МБ** (запас ~4 МБ на переупаковку).

Клиентская обработка (не серверная) — чтобы файл до 150 МБ не долетал до и так ограниченного `galaxyApp`-бэкенда (~512 МБ RAM, раздел 2), который вдобавок должен тянуть Puppeteer.

Итоговый PDF — окончательный производственный файл: идёт в SVG-превью, автоматически скачивается пользователю (`<a download>`/blob-URL) и грузится на Диск/в сделку. PDF как финальный формат (не AVIF) снимает риск несовместимости с печатным оборудованием/RIP-софтом.

**Живое превью PDF на изделии**: реализовано через `pdf.js` — страница рендерится в канвас (`getDocument`→`getPage(1)`→`render`→`toDataURL()`), вставляется в SVG `<image>` тем же способом, что `.png`/`.jpg`. Работает и для PDF, загруженного напрямую (≤52 МБ), и для итогового PDF после сжатия — превью и файл на Диске технически совпадают. Для `.ai`/`.eps` живого превью нет — клиентского способа их отрендерить не существует, остаётся плашка с расширением.

Edge case: если итоговый PDF всё же вышел бы за 52 МБ (маловероятно) — на Этапе 2 повторить с более агрессивным качеством AVIF; при повторной неудаче — тот же сценарий, что для векторных форматов.

Макет модалки — [`design/maket-6.html`](design/maket-6.html), финализирован, включает демо-кнопки для прогона обоих сценариев без реального крупного файла.

- комментарий
- ссылка/ID файла макета на Диске

## 7. Встраивание в Битрикс24

1. Пользователь открывает вкладку в карточке сделки (placement `CRM_DEAL_DETAIL_TAB`, подтверждён живым `GET /v1/placements/available`). Требует ключ `vibe_app_...`, привязку делает `POST /v1/placements/bind` (личным ключом `vibe_api_...` недоступно — см. раздел 2).
2. Gateway обменивает одноразовый `__init`-JWT на сессию и редиректит iframe на `<appUrl>/?placement=CRM_DEAL_DETAIL_TAB&member_id=...&placement_options=<JSON>&__init=...`. Frontend **не** получает `deal_id` напрямую — вместо этого парсит query-параметр `placement_options` (JSON-строка вида `{"ID":"42"}`, `URLSearchParams.get('placement_options')` + `JSON.parse`) и достаёт `ID` сделки оттуда. Через `BX24.placement.info()` контекст недоступен (SDK не видит его через handler-редирект).
3. Пользователь заполняет форму, загружает макет.
4. Frontend отправляет данные на BFF (`multipart/form-data`).
5. BFF проверяет идентичность пользователя: Gateway сам подставляет заголовок `X-Vibe-Authorization: Bearer vibe_session_...` на каждый запрос браузера к бэкенду — BFF пробрасывает его как `Authorization` в `GET /v1/me`, валидирует данные:
   - если `dealId` (из `placement_options`) передан — `PATCH /v1/deals/{id}`;
   - если отсутствует (открытие вне карточки сделки, если такой сценарий разрешён) — `POST /v1/deals`;
   - загружает файл на Диск (`POST /v1/files/upload`, до 52 МБ — раздел 4) и привязывает к сделке.
6. Возвращает статус операции на frontend.

Этапы 0–5 ниже относятся к Итерации 3+ (раздел 2.1) — кодирование начинается только после согласования дизайна в Итерациях 1–2.

### Этап 0 — Подготовка (сокращён)
- ✅ Инициализация монорепозитория: npm workspaces (`apps/frontend`, `apps/backend`, `packages/shared-types`), корневой ESLint (flat config) + Prettier, `tsconfig.base.json`. `npm install`, `npm run build`, `npm test`, `npm run lint` проходят чисто с корня.
  - `apps/frontend` — Vite + React 19 + TypeScript, Redux Toolkit и Ant Design подключены (плейсхолдер-экран), Vitest + React Testing Library.
  - `apps/backend` — Express + TypeScript, точка входа + `GET /api/health`, Vitest + supertest.
  - `packages/shared-types` — общие TS-типы формы заказа, потребляются напрямую из исходников (без отдельной сборки).
  - Стек тестирования обновлён по факту установки: **Vitest 4.x** (а не 2.x) — версия 2.x несовместима с Vite 8 (конфликт версий vite у vitest и `@vitejs/plugin-react`), потребовался апгрейд для единой версии vite в дереве зависимостей.
  - Node.js установлен вручную пользователем (v24.18.0, LTS на момент установки) — `winget`/скачивание файлов недоступны из песочницы инструментов агента (исходящий HTTPS для скачивания блокируется, но реестр npm — доступен).
- ✅ Фиксация списка UF-полей (раздел 6) в `packages/shared-types/src/index.ts` (`UF_FIELD_MAP`) — **коды полей плейсхолдерные** (`UF_CRM_PRODUCT_TYPE` и т.д.), реальные коды Битрикс24 присваиваются автоматически при создании полей на портале и должны быть подставлены до реализации Этапа 1.
- ✅ Определена JSON-структура размеров/количества — `SizeQuantities` (`Partial<Record<Size, number>>`) в `packages/shared-types`.
- ✅ Разведывательный проход по живой документации VibeCode и тестовому ключу — выполнен 2026-07-03, результаты и найденные расхождения документации с реальным API — раздел 2.
- ⏳ Создание ключа `vibe_app_...` (не `vibe_api_`, см. раздел 2) — ручной шаг на портале Битрикс24, не выполнен, нужен до начала Этапа 1 для placements/встраивания.

### Этап 1 — Backend / BFF
- `POST /api/order` с валидацией.
- Identity-middleware на основе `X-Vibe-Authorization` + `GET /v1/me` (без хранения/обновления токенов вручную).
- `createOrUpdateDeal(dealData, dealId?)`.
- `uploadFileToDisk(file, dealId)`.
- Обработка ошибок и логирование.

### Этап 2 — Frontend / SVG-конструктор
- Компоненты выбора (тип, ткань, уход за материалом, способ нанесения принта, 2 цветовые зоны, размеры/количество, доп. опции, зона размещения принта, комментарий).
- Загрузчик файлов с Drag&Drop и заглушкой для `.ai`/`.eps`.
- Модульное SVG-превью (группы: тело/силуэт, элементы отделки) с раздельной перекраской и наложением загруженного растрового макета в выбранной зоне печати.
- Сбор данных и отправка на backend.
- Задел на будущее (вне MVP): вид изделия сзади для размещения принта на спине.

### Этап 3 — Встраивание в Битрикс24
- Привязка placement `CRM_DEAL_DETAIL_TAB` через `POST /v1/placements/bind`.
- Проверка получения `deal_id` из query-параметров.

### Этап 4 — Тестирование
- Юнит/интеграционные тесты на каждую крупную функцию (Vitest + React Testing Library на фронтенде, Vitest + supertest на бэкенде), автопрогон (`npm test` в корне монорепозитория) после каждой правки.
- Проверка сценариев: создание сделки, обновление, загрузка разных форматов файлов, превышение лимита 115 МБ, работа внутри iframe.

### Этап 5 — Деплой и документация
- Деплой через `POST /v1/infra/servers/:id/deploy`.
- Обновление спецификации и README по итогам реализации (обязательное требование проекта — после каждой значимой итерации).

### Аудит безопасности
Проводится после согласования и реализации ~70% функциональности. Зоны внимания:
- валидация файлов (MIME по содержимому, лимиты, защита от path traversal в имени файла);
- корректность CSP (`frame-ancestors`) для работы во вложенном iframe;
- отсутствие утечек в логах (файлы, персональные данные из сделки).
